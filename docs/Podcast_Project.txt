1. PROJECT OVERVIEW
-------------------

Project Name:
EventSphere

Project Type:
Full Stack Event Management System

Goal:
Build a scalable event discovery and booking system where users can browse events, request or confirm bookings, and admins can manage events plus approve or reject paid bookings using real-world logic.

Core Product Thinking:
- Keep the system practical and explainable
- Focus on stable backend logic and consistent data flow
- Use role-based behavior instead of one-size-fits-all booking
- Prefer functional UI over over-designed visuals


2. HIGH-LEVEL SYSTEM ARCHITECTURE
---------------------------------

Frontend:
- React
- React Router
- Context for authentication state
- Axios for API communication

Backend:
- Node.js
- Express
- REST API routes
- Middleware-based auth and validation

Database:
- MongoDB Atlas

Main Data Flow:
User -> Frontend -> API request -> Express backend -> MongoDB Atlas
Admin -> Dashboard -> API request -> Express backend -> MongoDB Atlas

Authentication Flow:
- User logs in
- Backend validates credentials
- JWT is generated
- Token is stored in session storage on the frontend
- Protected APIs use the Bearer token

Booking Flow:
- User selects event
- Frontend sends booking request
- Backend checks seats, role, and duplicates
- Booking is stored in MongoDB
- Admin sees pending paid bookings and acts on them

7. FORGOT PASSWORD FLOW
-----------------------

The forgot-password flow is now documented and refined as a real 2-step verification flow.

Actual Flow:
Step 1:
- User opens Forgot Password
- User enters email
- Backend sends OTP to the registered email

Step 2:
- User enters OTP
- Backend verifies the OTP separately
- If OTP is correct, the user is allowed to continue

Step 3:
- User enters:
  - New Password
  - Confirm Password
- Backend updates the password in MongoDB

Important rule:
- OTP verification and password entry are not handled in the same user step anymore

Why this matters:
- It is closer to real-world auth design
- It reduces accidental password reset attempts without OTP proof
- It makes the flow easier to explain in interviews


8. EMAIL SERVICE EXPLANATION
----------------------------

Why Gmail App Password is used:
- Gmail blocks many direct login attempts from backend apps
- Normal Gmail passwords are not suitable for backend SMTP usage
- App Password provides a safer way for the backend to send OTP emails

How it is used:
- EMAIL_USER and EMAIL_PASS are stored in backend/.env
- Nodemailer uses these credentials to send verification and reset OTPs

Why it is practical:
- Easy to configure for student/demo projects
- Works with an existing Gmail account
- Good for OTP-based email flows

If email sending fails:
- The project has a console OTP fallback path for testing
- This prevents the whole auth system from breaking during development

10. JWT SECRET KEY EXPLANATION
------------------------------

JWT_SECRET is used to sign and verify authentication tokens.

Purpose:
- Confirms token authenticity
- Protects user sessions from simple tampering

Why it matters:
- If JWT_SECRET changes unexpectedly, old tokens become invalid
- If it is weak, token security is weaker

Recommendation:
- Use a long, random secret in production

13. BOOKING FLOW LOGIC
----------------------

This is the most important business rule in EventSphere.

Free Events:
- Automatically approved
- Seats are reduced immediately

Paid Events:
- Booking is created with status = pending
- Admin reviews it
- Admin can approve or reject

Why this logic is realistic:
- It mimics controlled-event platforms where paid or premium registrations may need moderation
- It allows admin oversight without blocking free/community participation

Additional booking rules:
- One booking has one status at a time
- Duplicate active bookings for the same user and event are blocked
- Seat availability is checked before booking
- Negative availableSeats is prevented

24. DYNAMIC SYSTEM DESIGN
-------------------------

EventSphere is no longer dependent on seed data for normal usage.

How it works now:
- Initial bootstrap data is used only to provide a ready starting state
- Admin, demo users, and starter events are seeded only when missing
- Existing database records are preserved and not overwritten on each server boot
- After startup, all new activity is driven by API calls and MongoDB writes

What is dynamic in the current system:
- Admin can create new events from the dashboard
- Users can register at any time using the live auth API
- New bookings are created from live event selections
- Booking approvals and rejections update live records
- Event lists and admin data are fetched from the backend, not from local mock arrays

Practical result:
- The system can keep growing without changing source code
- Seed data helps only with initial onboarding and demo setup
- Production behavior depends on the database, not on hardcoded objects


25. WHY NOT HARDCODE?
---------------------

Hardcoding users or events is a bad fit for a real system because:
- new records cannot be added naturally
- UI becomes disconnected from the real database
- admin actions do not scale
- deployment becomes brittle because data lives in code instead of storage

What was checked in this project:
- No hardcoded event list is used in the main event pages
- No hardcoded booking list is used in the admin dashboard
- No hardcoded user directory is used in the current UI

What is still intentionally static:
- category lists
- status label maps
- color mappings

These are UI metadata, not business data. They support rendering, but the actual users, events, and bookings come from MongoDB through APIs.

30. ARCHITECTURE CLASSIFICATION
-------------------------------

What kind of system is EventSphere?
- EventSphere is a full-stack MERN-style application
- Architecturally, it is a modular monolith
- The backend follows an MVC-inspired layered structure

Why "modular monolith" is the correct term:
- There is one backend application
- There is one main database
- Auth, events, and bookings are separate modules inside the same server
- The modules are separated by folders, but they are deployed together

Why it is not a microservices architecture:
- Microservices would split auth, events, bookings, notifications, and maybe payments into separate deployable services
- This project does not do that
- There is no API gateway, service discovery, inter-service messaging, or separate service database ownership
- For the current project size, team size, and academic scope, microservices would add unnecessary complexity

Why it is called "MVC-inspired" instead of strict MVC:
- Models exist clearly through Mongoose schemas
- Controllers handle business logic
- Routes act as the request-entry layer
- The "view" is not server-rendered inside Express
- The frontend React application handles the user interface separately

So the best viva answer is:
- "The backend is MVC-inspired and the overall system is a modular monolith with a separate React frontend."


31. WHY THIS FOLDER STRUCTURE?
------------------------------

Reason for separate folders:

`backend/`
- Keeps API logic, DB models, middleware, and server config isolated
- Makes backend debugging and deployment easier

`frontend/`
- Keeps UI, routes, pages, and API calls independent from server logic
- Allows the frontend to be built and deployed separately if needed

`docs/`
- Keeps project explanation, setup, and progress notes out of the source folders
- Makes the repository cleaner for GitHub, recruiters, and exam review

Why separate routes, controllers, models, middleware:
- routes define endpoints
- controllers contain logic
- models define data structure
- middleware handles reusable cross-cutting concerns like auth, validation, and DB checks

This separation improves:
- readability
- team collaboration
- debugging
- future scalability


32. TRICKY VIVA QUESTIONS AND STRONG ANSWERS
--------------------------------------------

Question:
- Why did you not use microservices?

Answer:
- Because the project scope is medium-sized and a modular monolith is more practical. It keeps deployment, debugging, and database consistency simpler while still allowing clear module separation.

Question:
- Why do you say this is not pure MVC?

Answer:
- Because the backend follows model-controller-route separation, but the view layer is handled by React on the frontend instead of Express templates. So it is better described as MVC-inspired.

Question:
- Why did you separate frontend and backend into different folders?

Answer:
- Because the frontend and backend have different responsibilities, dependencies, and build processes. This structure keeps the codebase cleaner and easier to maintain.

Question:
- Why are there separate middleware files?

Answer:
- Middleware prevents repeated logic. Auth checks, validation, DB readiness checks, and error handling can be reused across many routes without duplicating code.

Question:
- Why use MongoDB here?

Answer:
- Because this project benefits from flexible document storage, quick schema iteration, and easy relationship handling for users, events, and bookings in a MERN workflow.

Question:
- Why is this system scalable even though it is not microservices-based?

Answer:
- Because scalability first comes from clear module boundaries, database-driven flows, stateless APIs, and clean separation of concerns. Microservices are one scaling strategy, not the only one.

Question:
- What is the role of seed data in this project?

Answer:
- Seed data is only for initial bootstrap. The actual system is dynamic after startup, and new users, events, and bookings come from real API calls and database writes.
