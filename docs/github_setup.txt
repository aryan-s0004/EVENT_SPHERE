EVENTSPHERE - GITHUB SETUP GUIDE
================================

Repository Description (<= 320 characters):
EventSphere is a full-stack event management platform enabling users to discover and book events while allowing admins to manage and approve bookings. Built with React, Node.js, and MongoDB, it features real-time updates, role-based access, and scalable system design.


A. REPOSITORY CREATION
----------------------

1. Go to GitHub.
2. Click "New repository".
3. Repository name: `eventsphere`
4. Paste the repository description shown above.
5. Keep the repository empty for a clean first push.


B. BEFORE FIRST PUSH
--------------------

Checklist:
- confirm `.gitignore` exists
- confirm real `.env` files are ignored
- confirm `.env.example` is safe
- confirm `docs/` contains final documentation
- confirm frontend build passes
- confirm backend syntax is clean

Important:
- Do not upload real email passwords, JWT secrets, MongoDB URIs, or OAuth secrets
- Do not upload `backend/.env`
- Do not upload `frontend/.env`


C. INITIALIZE GIT
-----------------

Run from the project root:

```bash
git init
git branch -M main
```


D. DO NOT USE THIS
------------------

Avoid:

```bash
git add .
```

Reason:
- it hides what is being committed
- it increases the chance of pushing secrets
- it creates poor project history


E. PROFESSIONAL COMMIT STRATEGY (16 COMMITS)
--------------------------------------------

This is a recommended portfolio-style structure if you want a clean and explainable history.

Commit 1 - backend bootstrap and config
```bash
git add backend/package.json backend/package-lock.json backend/server.js backend/api backend/config
git commit -m "chore(backend): initialize server and configuration"
```

Commit 2 - backend models
```bash
git add backend/models
git commit -m "feat(backend): add core data models"
```

Commit 3 - backend middleware base
```bash
git add backend/middleware backend/utils/asyncHandler.js backend/utils/apiError.js
git commit -m "feat(backend): add middleware and shared error handling"
```

Commit 4 - backend auth module
```bash
git add backend/controllers/authController.js backend/routes/authRoutes.js backend/validators/authValidators.js backend/utils/generateOTP.js backend/utils/sendEmail.js backend/utils/fileToDataUri.js
git commit -m "feat(auth): implement registration, login, OTP, and profile flows"
```

Commit 5 - backend events module
```bash
git add backend/controllers/eventController.js backend/routes/eventRoutes.js backend/validators/eventValidators.js
git commit -m "feat(events): add event listing and admin event management"
```

Commit 6 - backend bookings module

F. ALTERNATIVE SMALLER COMMIT PLAN
----------------------------------

If you do not want 16 commits, use these 6 bigger commits:
- backend foundation
- backend features
- frontend foundation
- frontend features
- documentation
- root repo setup


F2. SIMPLE 5-COMMIT PLAN
------------------------

If you want a cleaner and easier student/team workflow, use these 5 commits only:

Commit 1 - backend
```bash
git add backend/
git commit -m "Add backend API and business logic"
```

Commit 2 - frontend
```bash
git add frontend/
git commit -m "Add frontend application and UI"
```

Commit 3 - documentation
```bash
git add docs/ README.md requirements.txt
git commit -m "Add project documentation and teammate setup guide"
```

Commit 4 - environment and repo config
```bash
git add .gitignore .env.example backend/.env.example frontend/.env.example package.json run_project.bat
git commit -m "Add environment examples and repository config"
```

Commit 5 - final review updates
```bash
git add backend/config/env.js backend/controllers/eventController.js backend/validators/eventValidators.js frontend/src/App.jsx frontend/src/pages/Login.jsx frontend/src/pages/AdminDashboard.jsx frontend/src/pages/Home.jsx
git commit -m "Refine production readiness and deployment setup"
```

Why this 5-commit plan is good:
- simple enough for teammates
- still keeps history readable
- separates app code from docs and config
- easy to explain during submission or viva


G. CONNECT THE REMOTE
---------------------

```bash
git remote add origin <repo-url>
```


H. PUSH TO GITHUB
-----------------

```bash
git push -u origin main
```


I. SECURITY RULES
-----------------

Never upload:
- `backend/.env`
- `frontend/.env`
- Gmail app passwords
- real JWT secrets
- real MongoDB credentials
- private Google OAuth secrets

Safe to upload:
- `.env.example`
- `backend/.env.example`
- source code
- docs
- README
- `.gitignore`


J. GOOD COMMIT PRACTICES
------------------------

- Keep backend and frontend commits separate where possible
- Do not mix docs, features, and config in one commit unless necessary
- Prefer meaningful commit messages over generic ones like "final update"
- Review changed files before every commit
- Build the frontend before the final push
- Recheck `.gitignore` before the first GitHub upload


K. FINAL GITHUB PRESENTATION TIPS
---------------------------------

For a recruiter-friendly repo:
- keep the README short and clear
- keep documentation inside `docs/`
- keep commit history readable
- avoid random temporary files
- use consistent naming

Best final message for this project:
- "This repository shows a modular full-stack Event Management System with clear auth flow, admin control, booking approval logic, and database-driven design."
